# obsidian-ingest

`raw/`에 새로 추가된 자료를 읽고 `wiki/`에 개체 페이지를 생성하거나 업데이트한다.

## 실행 절차

### 1. 처리 대상 자료 식별

`wiki/log.md`를 읽어 이미 `ingest` 처리된 파일명 목록을 파악한다.
`raw/` 디렉토리를 재귀적으로 스캔하여 log.md에 없는 파일을 처리 대상으로 선정한다.

처리 대상이 없으면 다음 메시지를 출력하고 종료:
```
처리할 새 자료가 없습니다. raw/에 파일을 추가한 후 다시 실행하세요.
```

처리 대상이 3개를 초과하면 다음 우선순위 기준으로 3개를 선택한다:

1. **범위가 넓은 문서 우선** — 개요·작동원리 문서가 먼저 있어야 이후 문서의 병합/신규 판단이 정확해진다
2. **독립 완결 문서 우선, 의존 문서 나중** — 다른 페이지 없이도 단독으로 완결되는 문서를 먼저 처리한다. 기존 페이지에 병합되거나 레퍼런스 성격의 문서는 나중에 처리한다
3. **위 기준이 동일하면 최근 추가 순** — 파일 수정 시각 기준

완료 리포트에 "미처리: N개 (다음 실행에서 처리)"를 명시한다.

### 2. 각 자료 처리

각 파일을 처리하기 전에 다음 원칙을 숙지한다:

> **작성 원칙**
> - **언어**: 기본적으로 한글로 작성한다. 번역 시 의미가 달라지거나 업계에서 영어로 통용되는 키워드(예: context window, fine-tuning, agentic loop)는 영어 원문을 그대로 사용한다.
> - 개념 이해에 필요한 정보는 빠뜨리지 않는다. 단, 같은 개념을 반복하는 중복 예제나 개념 이해에 도움이 되지 않는 예제는 생략한다.
> - 개념을 이해하는 데 핵심적인 코드 블록, 테이블, 패턴은 유지한다
> - 작업 패턴·예제 섹션 전체를 통째로 생략하지 않는다 — 섹션이 있다면 대표 예제 하나라도 남긴다
> - "너무 길다"는 이유만으로 섹션을 자르지 않는다. 불필요하다고 판단될 때만 생략한다
> - **설정·팁·단축키는 본문 어디에 묻혀 있어도 반드시 포함한다** — `settings.json` 키, 환경변수, 단축키, 비기본(non-default) 동작처럼 실용적 팁은 섹션 제목 없이 한 줄로 언급되더라도 wiki에 명시적으로 기술한다

처리 대상 파일별로 순서대로 실행:

#### 2-1. 자료 읽기
파일 전체를 읽는다. 이미지나 PDF처럼 직접 읽기 어려운 경우, 파일명·위치로 추론하지 않는다.
사용자에게 다음 형식으로 내용을 질문한다:

```
❓ 읽기 어려운 파일: [파일명]
이 파일의 주제/내용을 간략히 설명해 주세요.
```

사용자 답변을 받은 후 처리를 계속한다.

파일을 읽으면서 다음 항목을 별도로 메모한다 (본문 서술 중에 한 줄로 언급되는 경우도 포함):
- `settings.json` / `settings.local.json` 키
- 환경변수 (`ENV_VAR=value` 형태)
- 기본값이 아닌(non-default) 동작이나 숨겨진 설정
- "~를 설정하면", "~를 활성화하면" 류의 팁 문장

이 항목들은 2-7 작성 시 반드시 wiki 페이지에 포함시킨다.

#### 2-2. wiki/ 폴더 결정
자료 내용을 기반으로 가장 적합한 도메인 폴더를 결정한다. 기존 폴더에 구애받지 않고 내용 중심으로 판단한다.

주제 클러스터가 명확하면 서브폴더를 사용한다. 서브폴더를 새로 생성할 경우 CLAUDE.md 디렉토리 구조 테이블에 추가한다. 주제가 모호한 경우 서브폴더 없이 상위 폴더를 사용한다. `concepts/`는 여기서 선택하지 않는다 (2-5에서 처리).

#### 2-3. 파일명 결정
한글, 하이픈 구분. 파일명만 봐도 내용을 유추할 수 있도록 작성한다.
- 저자명/프로젝트명만으로 끝내지 않는다 (❌ `karpathy.md`)
- 핵심 주제를 포함한다 (✅ `karpathy-RNN의-비합리적-효과.md`)
- 예: `컨텍스트-윈도우-관리.md`, `LLM-양자화-가이드.md`, `claude-code-서브에이전트-병렬호출.md`
- 단, 업계에서 영어로 통용되는 고유명사(Claude Code, LLM, API 등)는 영문 유지

#### 2-4. 신규 생성 vs 병합 판단

새 페이지를 만들기 전에 기존 wiki 페이지와 동일 개념인지 확인한다.

**병합** (기존 페이지에 섹션 추가):
- 동일 주제를 다른 관점에서 다루는 경우 (가이드 vs 레퍼런스, 개요 vs 상세, 사용법 vs 스키마)
- 동일 도구·시스템의 추가 기능이나 확장 내용
- 기존 페이지의 특정 섹션을 심화하는 내용

**신규 파일 생성** (`1 파일 = 1 개체/개념`):
- 다른 페이지 없이도 독립적으로 완결되는 개체·개념
- 기존 페이지에 추가하면 제목(파일명)과 내용의 범위가 어긋나는 경우

#### 2-5. 핵심 개념·keywords 선정 및 concepts/ 판단
현재 자료의 핵심 개념을 식별하고 keywords를 확정한다.

**keywords 선정 기준**: 구현 상세(함수명, 파라미터, 옵션값)가 아닌 개념·패러다임·메커니즘 수준의 용어를 사용한다. 이 페이지의 핵심 개념을 빠짐없이 나열한다.

`grep "keywords:" wiki/**/*.md`로 기존 wiki 페이지의 keywords를 확인하고 다음 기준으로 판단한다.

- 기존 wiki 페이지의 keywords에 해당 개념이 있으면 → "관련 항목"에 링크 추가 예정
- 동일 개념이 3개 이상의 wiki 페이지 keywords에 등장하면 → concepts/ 페이지 생성
- concepts/ 페이지가 없는 개념은 아래 기준으로 생성 여부를 판단한다:
  - **생성**: 특정 도구·자료에 종속되지 않고 독립적으로 설명 가능한 개념
  - **생성 안 함**: 특정 도구의 기능 설명이거나 현재 자료에서만 의미 있는 내용

concepts/ 페이지를 새로 생성한 경우, 해당 개념이 keywords에 포함된 **기존 wiki 페이지들**의 `## 관련 항목` 섹션에 소급하여 링크를 추가한다:
```markdown
- [[wiki/concepts/개념명]]
```
**링크 대상 파일 존재 확인**: "관련 항목"에 추가할 각 링크에 대해 Glob으로 실제 파일이 존재하는지 확인한다. 존재하지 않는 파일은 링크 목록에서 제외한다.
이미 링크가 있는 페이지는 건너뛴다.

#### 2-6. 작성 전 승인 요청

페이지를 작성하기 전에 아래 형식으로 결정 내용을 요약하고 사용자 승인을 받는다:

```
📋 작성 계획

자료: [파일명]
액션: 신규 생성 / 기존 페이지 업데이트 ([기존 파일명])
경로: wiki/[폴더]/[파일명].md

제목: [페이지 제목]
keywords: [키워드1, 키워드2, ...]
요약: [2-3문장 요약]
concepts/ 생성: [개념명] / 없음

진행할까요?
```

수정 요청이 있으면 반영 후 다시 출력한다. `n`이면 해당 자료를 건너뛰고 완료 리포트에 "건너뜀: [파일명] (사용자 취소)"를 명시한다.

#### 2-7. 위키 페이지 작성
기존 파일이 있으면 내용을 읽고 업데이트. 없으면 새로 생성.

필수 필드를 반드시 포함:
```markdown
# 제목

**출처**: [URL 또는 raw/경로]
**날짜**: YYYY-MM-DD
**keywords**: [핵심개념1, 핵심개념2, ...]

## 요약
(2-3문장. 요약만 읽어도 이 페이지의 핵심을 완전히 파악할 수 있어야 한다. 개념이 무엇인지, 왜 중요한지, 어떻게 동작하는지를 압축한다.)

필수 필드 이후 내용은 자료 특성에 맞게 자유롭게 구성한다. 페이지 맨 끝에 `## 관련 항목` 섹션을 배치한다:
```markdown
## 관련 항목
- [[wiki/concepts/관련개념]]
```
**규칙**: 2-5에서 Glob으로 존재가 확인된 파일만 포함한다. 확인되지 않은 파일은 링크하지 않는다.

### 3. wiki/index.md 갱신

처리한 페이지를 해당 카테고리 섹션에 추가:
```markdown
- [[wiki/articles/karpathy-llm-wiki]] — LLM 기반 위키 구축 패턴
```
헤더의 페이지 수도 갱신한다.

### 4. wiki/log.md 기록 (필수, 생략 불가)

**이 단계는 반드시 실행해야 한다. 건너뛰면 스킬이 완료되지 않은 것으로 간주한다.**

처리한 파일마다 wiki/log.md 파일 끝에 추가한다:
```markdown
## [YYYY-MM-DD] ingest | 파일명
```
log.md의 기존 내용은 절대 수정하지 않는다. 오늘 날짜와 실제 처리한 파일명을 정확히 기입한다.

### 5. 완료 리포트 출력

4단계 log.md 기록이 완료된 후에만 아래 리포트를 출력한다.

```
[obsidian-ingest] 완료
처리한 자료: N개
생성된 페이지: 파일명1, 파일명2, ...
업데이트된 페이지: 파일명3, ...
생성된 개념 페이지: 개념명1, ...
생성된 폴더: wiki/books/ (사용자 승인)
건너뜀: 파일명4 (폴더 미승인)
log.md 기록: N건 추가
```
