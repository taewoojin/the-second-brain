# 문서 작성

코드가 아닌 문서가 산출물일 때 사용하는 스킬. 구조 합의 → 작성 → 검증 → 개선의 프로세스를 따른다.

## 핵심 원칙

- 구조 합의 전에 작성하지 않는다
- 문서는 코드와 다르게 태스크 분해 없이 구조 합의 후 바로 작성한다
- 모르는 사람도 이 문서만으로 이해할 수 있어야 한다 (완전성)

## 프로세스

```
1. 컨텍스트 수집 — 목적, 독자, 원본/참고자료 유무를 파악
2. 구조 설계 — 문서 구조를 제안하고 사용자 합의
3. 작성 — 합의된 구조에 따라 직접 작성
4. 검증 — 완전성, 용어 일관성, 가독성 확인
5. 개선 — 검증 결과를 바탕으로 수정
```

어느 단계에서든 이전 단계로 돌아갈 수 있다.
- 구조 변경 → 2단계(구조 설계)부터 재시작
- 범위/목적 변경 → 1단계(컨텍스트 수집)부터 재시작
- 용어/표현 수정 → 5단계(개선)에서 처리

## 1단계: 컨텍스트 수집

진입 시 아래 정보를 파악한다. **사용자의 지시에서 파악 가능한 정보는 다시 묻지 않는다.**

**필수:**
- **목적** — 이 문서가 왜 필요한가?
- **독자** — 누가 읽는가? (배경지식 수준)
- **범위** — 어디까지 다루는가?

**상황에 따라:**
- **원본/참고자료** — 기존 문서, 코드, 시스템 등 분석 대상이 있는가?
- **형식 제약** — 특정 템플릿, 저장 위치, 파일 형식

**질문 방식:**
- 한 번에 하나의 질문
- 객관식 선호, 개방형도 가능

**언어 규칙:**
- 기본적으로 한글로 작성
- 고유명사, 한글로 표현했을 때 의미가 애매한 용어는 영어로 표기
- 함축적이거나 애매한 단어는 의미를 풀어서 작성 (예: "퍼포먼스적 동의" → "형식적 동의")

## 2단계: 구조 설계

컨텍스트 수집이 끝나면 문서 구조를 제안한다.

**프로세스:**
1. 원본/참고자료가 있으면 먼저 전체를 읽고 파악
2. 문서 구조(섹션, 계층, 파일 분할 여부)를 제안
3. 사용자 합의 후 다음 단계로 진행

**구조 설계 원칙:**
- 정보는 사용되는 맥락에 배치한다 (별도 참조 섹션으로 분리하지 않는다)
- 각 섹션이 독립적으로 읽힐 수 있어야 한다
- 복잡도에 비례하여 분량을 조절한다 (간단한 내용을 길게 쓰지 않는다)

**문서가 길어질 경우 (200줄 이상 예상):**
- 인덱스(README) + 주제별 하위 문서 + 퀵레퍼런스 구조를 제안
- 인덱스는 네비게이션만 담당하고 내용을 포함하지 않는다
- 각 하위 문서 상단에 TL;DR (3-5줄)을 포함한다
- 분할 후 개별 문서가 TL;DR을 제외하고 본문 50줄 미만이 되면 관련 문서와 병합한다 (문서 하나가 너무 짧으면 독립 문서로서의 가치가 없다)

**합의 전에 작성하지 않는다.** 구조가 확정되어야 작성을 시작한다.

## 3단계: 작성

합의된 구조에 따라 문서를 작성한다.

**작성 원칙:**
- 이 문서만으로 주제를 모르는 사람도 이해할 수 있어야 한다 (완전성)
- 재구성일 경우 원본 정보를 누락하지 않는다 (요약이 아닌 재구성)
- 문서 간, 섹션 간 내용이 중복되지 않아야 한다
- 코드 블록, 테이블, 다이어그램은 원본 그대로 유지한다
- 이미 명확한 것을 다시 설명하지 않는다 (불필요한 메타 설명 금지)

**다이어그램:**
- ASCII 다이어그램에서 박스(┌─┐)와 한글이 함께 사용되면 렌더러마다 정렬이 깨진다 → Mermaid 사용
- 박스 없이 화살표와 텍스트만으로 구성된 다이어그램은 ASCII가 더 간결하다 → ASCII 유지
- Mermaid 노드 안에서 줄바꿈은 백틱+쌍따옴표 문법을 사용한다

**인덱싱:**
- 원본에서 분할한 경우, 원본의 섹션 번호를 그대로 가져오지 않는다
- 각 문서는 자체적인 제목 체계를 가진다

## 4단계: 검증

작성이 완료되면 아래 항목을 검증한다.

**완전성:**
- 원본/참고자료가 있는 경우, 모든 섹션이 빠짐없이 포함되었는지 확인
- 각 문서가 독립적으로 읽혀서 이해 가능한지 확인

**용어 일관성:**
- 설명 없이 사용된 전문 용어나 약어가 없는지 확인
- 같은 개념에 다른 표현이 혼용되지 않는지 확인
- 한글 번역이 혼란을 줄 수 있는 용어는 의미를 풀어서 작성했는지 확인
- 타이틀에 사용된 영어 용어는 한글 번역 + 괄호 영어 형식인지 확인 (예: "경고 신호(Red Flags)")

**가독성:**
- 다이어그램이 렌더링 환경에서 깨지지 않는지 확인
- 문서 간 중복 내용이 없는지 확인
- 불필요한 메타 설명이나 원본 구조의 잔재(섹션 번호 등)가 남아있지 않은지 확인

## 5단계: 개선

검증에서 발견된 문제를 수정한다.

**프로세스:**
1. 검증 결과를 사용자에게 보고 (발견된 항목 목록)
2. 수정 방향을 제안하고 사용자 합의
3. 수정 실행
4. 수정 후 해당 항목을 재검증

**수정하지 않는 것:**
- 사용자가 승인한 구조 자체를 임의로 변경하지 않는다
- 검증 범위를 넘어서는 "개선"을 추가하지 않는다 (요청된 것만 수정)

## 합리화 방지

| 합리화 | 현실 |
|--------|------|
| "이미 작성했으니 부분 수정으로 충분하다" | 구조가 잘못되면 부분 수정은 미봉책이다. 돌아가라. |
| "구조 합의 없이 먼저 작성하고 보여주겠다" | 합의 없는 작성은 재작업을 만든다. 구조 먼저. |
| "검증은 나중에 한번에 하겠다" | 작성 후 즉시 검증하지 않으면 누락을 놓친다. |
| "이 정도면 독자가 이해할 것이다" | 추측이 아닌 검증. 모르는 사람 기준으로 확인하라. |
| "용어가 업계에서 통용되니 설명 불필요하다" | 독자의 배경지식을 가정하지 마라. 풀어서 쓰라. |
| "분량이 너무 길어지니 일부를 생략하겠다" | 완전성이 우선이다. 길면 구조를 분할하라. |
