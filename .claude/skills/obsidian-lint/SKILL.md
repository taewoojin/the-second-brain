# obsidian-lint

`wiki/`를 읽고 LLM만이 할 수 있는 의미론적 건강성 점검을 수행한다.
기계적으로 감지할 수 있는 것(링크 깨짐, 필드 누락)이 아닌, **판단이 필요한 이슈**에 집중한다.

## 점검 항목

### 1. 페이지 간 모순
서로 다른 페이지에서 같은 개념을 상반되게 설명하는 경우.
예: A 페이지에서는 "X는 Y다", B 페이지에서는 "X는 Z다".

### 2. 오래된 주장
최신 자료로 대체됐거나 현재 상식과 어긋나는 내용.
예: 더 나은 방법이 등장했는데 이전 방법을 권장하는 페이지.

### 3. 개념 공백
여러 페이지에서 언급되지만 독립 페이지가 없는 중요 개념.
예: "attention mechanism"이 세 곳에서 등장하지만 `wiki/concepts/attention-mechanism.md` 없음.

### 4. 누락된 교차참조
A 페이지와 B 페이지가 밀접히 연관되어 있는데 서로 `[[링크]]`하지 않는 경우.

### 5. 고립된 페이지
다른 어떤 페이지에서도 참조되지 않는 페이지. 위키 탐색 중 닿을 수 없는 섬.

### 6. 데이터 공백
페이지 내용이 불완전하거나 모호하여, 웹 검색으로 보강하면 크게 나아질 부분.

### 7. 동일 개념 분산
같은 개념을 다룬 페이지가 두 개 이상 존재하는 경우. `1 파일 = 1 개체/개념` 원칙 위반.
예: "가이드"와 "레퍼런스"처럼 관점만 다르고 주제가 동일한 페이지 쌍 → 병합 권장.

## 실행 절차

### 1. 위키 전체 읽기

`wiki/` 아래 모든 `.md` 파일(index.md, log.md 제외)을 읽는다.

### 2. 각 점검 항목별 분석

위 6가지 항목을 순서대로 검토한다.
실제 내용을 비교·판단해서 이슈를 도출한다. 추측이 아닌 근거 있는 이슈만 기록한다.

### 3. 리포트 출력

```markdown
## [obsidian-lint] 리포트 — YYYY-MM-DD

### 페이지 간 모순
- `wiki/a.md` vs `wiki/b.md`: [모순 내용 설명]

### 오래된 주장
- `wiki/c.md`: [무엇이 오래됐는지, 왜 그렇게 판단했는지]

### 개념 공백
- `개념명`: [어떤 페이지들에서 언급됐는지] → 새 페이지 생성 권장

### 누락된 교차참조
- `wiki/d.md` ↔ `wiki/e.md`: [연관 이유]

### 고립된 페이지
- `wiki/f.md`: 어떤 페이지에서도 참조되지 않음

### 데이터 공백
- `wiki/g.md`: [어떤 정보가 부족한지] → 웹 검색 권장

### 동일 개념 분산
- `wiki/h.md` + `wiki/i.md`: [왜 같은 개념인지] → 병합 권장

### 조사 권장 질문
- [위키를 성장시킬 새 탐구 방향 제안]

---
총 이슈: N개
이슈 없음: ✓ (이슈가 0개인 경우)
```

### 4. wiki/log.md에 기록 (append only)

```markdown
## [YYYY-MM-DD] lint | 총 N개 이슈 (모순:N, 오래됨:N, 개념공백:N, 교차참조:N, 고립:N, 데이터공백:N, 분산:N)
```

이슈가 0개면:
```markdown
## [YYYY-MM-DD] lint | 이슈 없음 ✓
```
