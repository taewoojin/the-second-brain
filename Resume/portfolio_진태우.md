## Project 1. 모듈화 아키텍처 설계
*단일 모듈 iOS 앱을 Clean Architecture 기반 멀티 모듈로 점진 전환*

### 배경 / 문제

크몽 iOS 앱은 단일 모듈 구조로 운영되고 있었습니다. SwiftUI를 적극적으로 사용하게 되면서 Preview 활용이 필요해졌고, 서비스 성장과 함께 다음 문제가 누적되었습니다:

1. **빌드 시간 증가** — 단일 모듈이라 코드 변경 시 전체 앱을 빌드해야 하며, SwiftUI Preview도 사실상 사용 불가
2. **의존성 방향 제약 없음** — Clean Architecture 원칙이 컴파일 수준에서 보장되지 않아 레이어를 건너뛰는 코드 발생 가능. 싱글톤 객체들이 구현체를 직접 참조하는 등 격리 테스트가 어려운 구조

### 내가 한 것

모듈 구조는 팀(4명) 공동 설계. 이후 순환 의존성 해결과 점진적 이관 작업을 리딩하며 가장 많은 이관을 수행.

**핵심 기술적 의사결정:**

**1. 모듈 분할 방식 — Layered + Domain 혼합**

Layered(계층 단위)는 의존성 방향을 강제하지만 도메인 간 격리가 불가하고, Feature-Based(기능 단위)는 독립 개발이 가능하지만 공통 코드가 중복됩니다.

→ **혼합 적용**: 기술 계층(Feature/Domain/Data)으로 모듈을 나누고,
Domain 내부에서 21개 비즈니스 도메인을 분리하여 두 접근의 장점을 결합

```
App → Feature → Domain ← Data
             ↘ DesignSystem → UIFoundation

Foundation, ThirdParty: 전 레이어에서 공통 사용
```

→ **단계적 전략**: 1단계로 레이어별 모듈을 분리하여 의존성 방향을 강제하고, 코드 이관 완료 후 2단계로 Feature 내 화면별 타겟 분리를 통해 Preview 및 증분 빌드 속도를 개선하기로 결정

**2. Tuist 선택**
모듈화 도구로 SPM, CocoaPods 기반 수동 구성, Tuist를 검토.
- Swift Manifest로 프로젝트 구조를 정의 → .xcodeproj Git 충돌 해소, 팀원 간 설정 불일치 방지
- 환경별 설정(Debug/Release/Staging)을 코드로 중앙 관리
- 당시 국내 iOS 커뮤니티에서 적용 사례와 자료가 가장 풍부

**3. Static Library/Framework 선택**
Static/Dynamic Library/Framework의 다양한 조합을 실험하여 검증.
- Dynamic + Static 혼합 시 구성에 따라 Static 코드가 각 Dynamic Framework에 중복 포함될 수 있고,
  이 경우 싱글턴 인스턴스가 서로 다른 인스턴스로 인식되는 문제 확인
- 팀원 모두 모듈화 경험이 없었기에, 이런 혼합 구성 리스크를 피해 전체 Static으로 통일

### 해결 과제

**순환 의존성 해결:**
단일 모듈에서는 파일 간 상호참조가 문제없이 빌드되지만,
모듈을 분리하면 순환 import가 컴파일 에러로 드러남.
예) 인증 관리자 ↔ 토큰 관리자, 인증 관리자 ↔ 분석 관리자 등
Singleton Manager 간 직접 참조가 얽혀있던 것을 Protocol로 추출하여 의존성 방향을 정리.

**운영 중 서비스의 점진적 이관:**
전체 코드를 한번에 모듈로 옮기면 서비스 안정성 리스크가 큼.
화면 단위로 이관을 진행하되, 먼저 해당 화면이 의존하는 순환 참조를 끊어낸 후
코드를 각 모듈(Feature/Domain/Data)에 맞게 이동하는 순서로 진행.

### 결과

- **의존성 방향 컴파일 강제**: 잘못된 참조 원천 차단, Clean Architecture 위반 불가
- **테스트 가능 구조 확보**: Protocol 추상화와 DI 적용으로 UseCase/ViewModel 격리 테스트 및 Mock 주입 가능
- **SwiftUI Preview 실행 가능**: 모듈 단위 빌드로 Preview 활용 가능
- 전체 3,386개 파일 중 805개(24%) 모듈화 완료, 점진적 이관 중

## Project 2. Claude Code 활용 환경 구축
*AI가 프로젝트 규칙을 이해하고 코드를 생성하는 개발 파이프라인 설계*

### 배경 / 도입 과정

4명의 iOS 팀에서 개발 생산성을 높이기 위해 Claude Code를 도입했습니다.
본인이 먼저 피처 작업에 적용하여 생산성 향상을 검증한 후, 규칙 체계와 커맨드를 정비하여 팀에 공유했습니다.
가치 기반 자율 채택 방식으로 팀 전원이 핵심 도구를 사용하게 되었고,
개발 속도가 올라가면서 발생한 새로운 병목(PR 리뷰 부담 증가)을 AI 리뷰 에이전트로 해결하는 등 점진적으로 확장해갔습니다.

### 내가 만든 것

**1. CLAUDE.md 기반 팀 규칙 시스템**

아키텍처 규칙, 네이밍 컨벤션, 의존성 규칙을 체계화하여 AI가 프로젝트 규칙을 자동 준수하도록 구성. 처음에는 상세하게 작성했으나, 프로젝트 고유 특성만 명시하고 나머지는 AI의 기존 지식에 위임하는 것이 더 효과적이었습니다.

**2. 요구사항 → 코드 생성 파이프라인**

```
/plan (명세·설계 + 마일스톤 분할) → /plan-refine (설계 검증) → /plan-work (코드 생성)
    → /commit (자동 커밋) → /pr (PR 생성) → /pr-review (AI 리뷰)
```

핵심은 `/plan` → `/plan-refine` → `/plan-work` 3단계 워크플로우입니다:
- **`/plan`**: 자연어 요구사항을 기술 명세로 변환하고 구현 마일스톤을 분할
- **`/plan-refine`**: insight 에이전트가 설계를 반복 검증하여 아키텍처 위반·누락을 사전 탐지
- **`/plan-work`**: 검증된 명세를 기반으로 마일스톤 단위로 코드 생성·테스트·커밋

이 파이프라인으로 Project 3(서비스 상세 마이그레이션)에서 21개 마일스톤을 체계적으로 관리하며, 요구사항부터 코드까지 일관된 컨텍스트를 유지할 수 있었습니다.

**3. AI PR 리뷰 에이전트**

팀원들이 Claude Code로 개발 속도가 올라가자 PR 리뷰 요청이 2배 이상 빨라지면서 리뷰어 부담이 증가하는 새로운 병목이 발생했습니다.

`pr-reviewer` 에이전트를 구축하여:
- PR 생성 시 AI가 1차 리뷰를 자동 수행하여 GitHub에 코멘트 제출
- 작성자가 AI 코멘트를 먼저 반영 → 리뷰어는 AI 코멘트 기반으로 집중 리뷰

**4. 커스텀 커맨드 — `/endpoint-generator` 등 9개**

Swagger 스펙 기반 Endpoint → Swift 코드 자동 변환으로 **Endpoint 작성 시간 약 30분 → 5분으로 단축 (83% 감소)**. 그 외 `/commit`, `/report`, `/teach` 등 반복 작업 자동화 커맨드 구축.

### 결과

- **Endpoint 작성 83% 단축**: Swagger → Swift 코드 자동 변환 (30분 → 5분)
- **리뷰 효율 개선**: AI 1차 리뷰로 셀프 리뷰 효과 → 리뷰어 부담 경감
- **팀 규칙 자동 준수**: CLAUDE.md 기반으로 AI가 팀 컨벤션을 일관 적용
- **자연스러운 팀 채택**: 가치 기반 자율 채택으로 팀 전원이 핵심 도구(`/pr`, `pr-reviewer`) 활용

## Project 3. 서비스 상세 화면 SwiftUI 마이그레이션
*오랜 기간 누적된 God Object를 책임 분리하고 SwiftUI로 전면 재구축*

### 배경 / 문제

서비스 상세(GigDetail)는 크몽 앱의 핵심 화면으로, 오랜 기간 누적된 기술 부채가 유지보수를 심각하게 저해하고 있었습니다:

- **God Object화된 핵심 파일** — 메인 ViewController 2,033줄, ViewModel 1,137줄로 단일 파일에 과도한 책임이 집중
- **Storyboard/XIB 기반 UI (44개 파일)** — XML 기반으로 코드 리뷰가 어렵고, Git 충돌이 빈번

### 내가 한 것

전체 마이그레이션 수행. Project 2에서 구축한 AI 파이프라인(/plan → /plan-work)으로 21개 마일스톤 관리.

**핵심 기술적 의사결정:**

**1. God Object 분해 전략 — 이중 기준 적용**

2,033줄 ViewController의 책임이 두 축으로 얽혀있었습니다:
- **UI 섹션**: 갤러리, 패키지, 리뷰, 하단 바 등 화면 영역별 상태 관리
- **비즈니스 액션**: 주문, 문의, 전화 등 여러 UI 섹션에서 공통으로 트리거되는 로직

단순히 한 축으로만 분리하면 중복이나 분산 문제가 발생합니다.

→ **UI 섹션별로 View/ViewModel을 분리하고, 2개 이상의 섹션에서 공유하는 비즈니스 로직은 별도 클래스로 추출**하여 두 문제를 동시에 해결.

```
[Before]
ViewController (2,033줄) ─── 갤러리 UI + 패키지 UI + 리뷰 UI + 주문 로직 + 문의 로직 + 전화 로직
ViewModel (1,137줄)      ─── 모든 섹션의 상태 + 모든 비즈니스 로직

[After]
GigDetailView ─┬─ GalleryViewModel (갤러리 UI 상태)
               ├─ PackageViewModel (패키지 UI 상태)
               ├─ ReviewViewModel (리뷰 UI 상태)
               ├─ BottomBarViewModel (하단 바 UI 상태)
               ├─ InfoViewModel (기본 정보 UI 상태)
               ├─ SellerViewModel (판매자 UI 상태)
               │
               ├─ OrderHandler (주문 로직 — 갤러리, 하단 바에서 공유)
               ├─ InquiryHandler (문의 로직 — 하단 바, 판매자에서 공유)
               ├─ CallHandler (전화 로직 — 하단 바, 판매자에서 공유)
               │
               └─ GigDetailContext (공유 데이터: 서비스 상세 데이터, 사용자 정보, 라우터)
```

**2. 공유 데이터 설계 — Context 객체**

분리된 ViewModel과 비즈니스 로직 클래스가 서비스 상세 데이터, 사용자 정보, 라우터 등을 공유해야 했습니다. 각각 주입하면 의존성이 복잡해지므로, 불변 Context 객체에 공유 데이터를 모아 단일 진입점으로 전달. 하위 객체 간 결합도를 최소화하면서 필요한 데이터 접근을 보장.

**3. 운영 중 안전한 전환**

기존 UIKit 코드를 수정하지 않고 SwiftUI 버전을 완전히 새로 구현. Feature Flag에 따라 Route 레벨에서 전환되는 View만 교체하는 방식으로, 두 버전이 ViewModel을 공유하지 않아 상호 영향 없이 동시 운영.

### 해결 과제

**추출 경계 결정 — "ViewModel에 남길 것 vs 별도 클래스로 추출할 것":**

God Object에서 로직을 분리할 때 적용한 원칙: **2개 이상의 UI 섹션에서 호출되는 로직은 별도 클래스로 추출**. 예를 들어 주문 로직은 상단 갤러리와 하단 바 모두에서 트리거되므로 분리하고, 리뷰 데이터 로딩처럼 단일 섹션에서만 사용하는 로직은 해당 Child ViewModel에 유지.

**분해 후 테스트 전략:**

Context를 통한 의존성 주입 구조 덕분에 Mock 주입이 용이하여, 비즈니스 로직 단위 테스트와 통합 시나리오 테스트를 작성.

### 결과

- **God Object 분해**: 2,033줄 단일 파일 → UI 섹션별 ViewModel 6개 + 공유 비즈니스 로직 3개 + Context 1개 (각 파일이 단일 책임)
- **XIB 44개 제거**: Storyboard/XIB → 순수 Swift 코드로 전환하여 코드 리뷰와 diff 정상화
- **테스트 가능 구조 확보**: Context 기반 DI로 비즈니스 로직/ViewModel 단위 테스트 작성 가능
- **AI 파이프라인 실증**: Project 2의 /plan → /plan-work 워크플로우로 21개 마일스톤을 체계적으로 관리
