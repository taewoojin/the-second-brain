# 에이전트 우선 개발 — 하네스 엔지니어링의 Codex 활용 사례

**출처**: https://openai.com/ko-KR/index/harness-engineering/ | **날짜**: 2026-04-14  
**원저자**: Ryan Lopopolo, 기술 스태프 멤버 (OpenAI)

## 요약
OpenAI 하네스(Harness) 엔지니어링 팀이 5개월 동안 수동 코드 한 줄도 작성하지 않고 Codex 에이전트만으로 내부 소프트웨어 제품을 구축한 사례 연구다. 3명의 엔지니어가 약 100만 라인의 코드와 1,500개의 PR을 생성했으며, 엔지니어의 역할이 "코드 작성"에서 "에이전트가 일할 수 있는 환경 설계"로 전환되었다는 핵심 교훈을 담고 있다.

## 핵심 수치
- 기간: 5개월
- 팀 규모: 3명 → 7명으로 성장
- 코드 규모: ~100만 라인
- PR: ~1,500개 (엔지니어 1인당 하루 평균 3.5개)
- 사람이 작성한 코드: 0줄

## 에이전트 역할 재정의

**Before**: 엔지니어가 코드 작성 → Claude/Codex가 리뷰
**After**: 엔지니어가 환경 설계 → 에이전트가 코드 작성/리뷰/배포

사람은 다음만 담당:
- 업무 우선순위 결정
- 사용자 피드백을 수용 기준으로 전환
- 에이전트가 막힐 때 환경 개선 신호로 해석

## 핵심 교훈 1: AGENTS.md를 백과사전이 아닌 목차로

**실패 패턴 — 거대한 단일 AGENTS.md**:
- 컨텍스트가 희소 리소스인데 불필요한 내용으로 낭비
- "모든 것이 중요"하면 정작 중요한 것이 없어짐
- 낡은 규칙의 무덤으로 변함 (에이전트가 무엇이 현재도 사실인지 모름)

**해결책**: 짧은 AGENTS.md(~100줄)는 목차 역할, 상세 내용은 구조화된 `docs/` 디렉토리에 분리
```
AGENTS.md  ← 맵 역할 (~100줄)
ARCHITECTURE.md
docs/
├── design-docs/index.md
├── exec-plans/active/ completed/
├── generated/db-schema.md
├── product-specs/
└── references/  ← llms.txt 파일 등
```

**점진적 공개**: 에이전트가 처음부터 부담 없이 작고 안정적인 진입 지점에서 시작해 다음 단계로 이동.

## 핵심 교훈 2: 에이전트 가독성 최적화

에이전트에게 보이지 않는 것은 존재하지 않는 것과 같다:
- Google Docs, Slack 메시지, 사람들의 머릿속 지식 → 에이전트 접근 불가
- 저장소 내 버전 관리 아티팩트(코드, 마크다운, 스키마, 실행 계획)만 접근 가능

실질적 조치:
- Chrome DevTools Protocol 연결 → Codex가 UI 직접 테스트/검증 가능
- 워크트리별 로컬 관측 가능성 스택 제공 → LogQL/PromQL로 로그/메트릭 직접 쿼리
- Slack 토론에서 나온 아키텍처 패턴 → 즉시 저장소 마크다운으로 기록

## 핵심 교훈 3: 아키텍처와 취향을 기계적으로 강제

문서화만으로는 일관성 유지 불가 → **린터와 구조 테스트로 불변 조건 강제**

- 레이어드 도메인 아키텍처: `Types → Config → Repo → Service → Runtime → UI`
- 교차 관심사(인증, 텔레메트리 등)는 Providers 인터페이스 하나로만 진입
- 맞춤형 린터로 구조화된 로깅, 명명 규칙, 파일 크기 제한을 정적으로 강제
- 린트 에러 메시지에 수정 지침 포함 → 에이전트 컨텍스트에 직접 주입

## 핵심 교훈 4: 엔트로피와 가비지 컬렉션

에이전트가 불균일한 패턴도 복제 → 시간이 지나면 드리프트 발생

**실패 시도**: 매주 금요일 팀이 수동으로 "AI 슬로프" 정리 (확장 안 됨)

**해결책**: "황금 원칙"을 코드로 인코딩 + 정기적 정리 에이전트 운영
- Codex 백그라운드 작업이 정기적으로 품질 등급 업데이트, 리팩토링 PR 자동 생성
- 대부분 1분 이내 검토 후 자동 병합
- 기술 부채 = 고금리 대출 → 조금씩 꾸준히 상환

## 에이전트 자율성의 현재 수준

한 번의 프롬프트로 Codex가 수행:
1. 코드베이스 현재 상태 검증
2. 버그 재현 + 영상 녹화
3. 수정사항 구현 및 앱 실행 검증
4. PR 생성 + 피드백 응답
5. 빌드 실패 감지 및 수정
6. 판단 필요 시만 사람에게 에스컬레이션
7. 변경사항 병합
