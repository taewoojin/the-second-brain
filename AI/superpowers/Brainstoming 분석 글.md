       1. 개요

       Brainstorming 스킬은 Superpowers 플러그인의 설계 단계 게이트키퍼다. AI 에이전트가 코드를 작성하기 전에 반드시 거쳐야 하는 협업적 설계 프로세스를 정의한다. "아이디어를 완성된 설계로 변환한다"는 것이 핵심 목적이며, 구현은 이 스킬의 범위 밖이다.

       파일 구조

       skills/brainstorming/

       2. 설계 철학과 핵심 원칙

       2.1 HARD-GATE 패턴

       SKILL.md에서 가장 눈에 띄는 설계 결정은 <HARD-GATE> 태그로 감싼 절대적 금지 규칙이다:

       ▎ 설계를 제시하고 사용자가 승인하기 전까지 어떤 구현 스킬도 호출하지 말 것, 코드를 작성하지 말 것, 프로젝트를 스캐폴딩하지 말 것.

       이것은 AI 에이전트가 "간단한 것은 바로 구현하면 되지"라는 판단을 내리는 것을 원천 차단한다. "이건 설계할 필요 없을 만큼 단순하다"는 안티패턴으로 명시적으로 경고하며, todo 리스트 같은 작은 프로젝트도 예외가 아니라고 못 박는다.

       설계 의도: AI 에이전트는 확인되지 않은 가정(unexamined assumptions) 위에 코드를 쌓는 경향이 있다. 이 게이트는 그 경향을 구조적으로 차단한다.

       2.2 단일 질문 원칙

       ▎ 한 번에 하나의 질문만 한다.

       AI가 한꺼번에 여러 질문을 쏟아내면 사용자는 피로감을 느끼고 대충 답하게 된다. 가능하면 객관식(multiple choice)을 선호하도록 안내하여 사용자 응답의 부담을 낮춘다.

       2.3 YAGNI (You Aren't Gonna Need It)

       설계 원칙에 "YAGNI를 무자비하게 적용하라(ruthlessly)"고 명시한다. 모든 설계에서 불필요한 기능을 제거하는 것이 기본 자세다.

       2.4 격리와 명확성을 위한 설계

       시스템을 작은 단위로 분해하되, 각 단위가 하나의 명확한 목적을 가지고 잘 정의된 인터페이스로 소통하며 독립적으로 이해/테스트 가능해야 한다고 강조한다. 이 원칙은 AI 에이전트의 특성과도 연결된다 -- AI는 한 번에 컨텍스트에 담을 수 있는 코드에 대해 더 정확한 추론을
        하므로, 파일이 커지면 그 자체가 분리 신호라고 안내한다.

       3. 프로세스 흐름 (9단계 체크리스트)

       스킬은 반드시 순서대로 완료해야 하는 9단계 체크리스트를 정의한다:

       4. 프로젝트 컨텍스트 탐색
       5. 시각적 동반자 제안 (시각적 질문이 예상되는 경우)
       6. 명확화 질문 (한 번에 하나씩)
       7. 2~3가지 접근 방식 제안 (트레이드오프 + 추천)
       8. 설계 제시 (섹션별, 복잡도에 비례하는 분량)
       9. 설계 문서 작성 → docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md
       10. 스펙 리뷰 루프 (서브에이전트 디스패치, 최대 5회 반복)
       11. 사용자 리뷰 게이트
       12. writing-plans 스킬로 전환

       종착 상태(terminal state)는 writing-plans 스킬 호출이다. 다른 구현 스킬(frontend-design, mcp-builder 등)으로 직접 전환하는 것은 금지된다. 이 제약은 brainstorming -> writing-plans -> executing-plans로 이어지는 Superpowers의 파이프라인 구조를 강제한다.

       대규모 프로젝트 분해

       스코프 판단이 중요하다. 요청이 여러 독립적인 하위 시스템을 포함하면(예: "채팅, 파일 저장소, 결제, 분석이 있는 플랫폼"), 세부 질문으로 들어가기 전에 먼저 분해를 제안해야 한다. 각 하위 프로젝트는 자체적인 스펙 -> 계획 -> 구현 사이클을 거친다.

       14. Visual Companion 시스템

       5.1 아키텍처

       Visual Companion은 AI 에이전트가 브라우저를 통해 사용자에게 목업, 다이어그램, 시각적 선택지를 보여주는 시스템이다. 제로 의존성(zero-dependency) Node.js 서버로 구현되어 있으며, npm install 없이 동작한다.

       ┌─────────────┐     파일 감시      ┌─────────────┐     WebSocket     ┌─────────────┐
       │  AI 에이전트  │ ──HTML 파일 작성──> │  server.js   │ ──reload 알림──> │   브라우저    │
       │  (터미널)     │                    │  (Node.js)   │ <──클릭 이벤트── │  (사용자)     │
       │              │ <──.events 읽기──  │              │                  │              │
       └─────────────┘                    └─────────────┘                  └─────────────┘

       통신 방식이 독특하다:
       - AI -> 브라우저: 파일 시스템을 매개로 한다. AI가 HTML 파일을 screen_dir에 쓰면, 서버가 fs.watch로 감지하고 WebSocket을 통해 브라우저에 reload 메시지를 보낸다.
       - 브라우저 -> AI: 사용자 클릭은 WebSocket으로 서버에 전달되고, 서버가 .events 파일에 JSONL로 기록한다. AI 에이전트는 다음 턴에 이 파일을 읽는다.

       이 설계는 AI 에이전트가 HTTP API를 호출할 수 없는 환경에서도 동작하도록 파일 시스템을 공유 매개체로 사용한 것이다.

       5.2 server.js 상세 분석

       WebSocket 직접 구현: ws 라이브러리 대신 RFC 6455를 직접 구현했다. computeAcceptKey, encodeFrame, decodeFrame 함수가 프로토콜 핸드셰이크와 프레임 인코딩/디코딩을 처리한다. 외부 의존성 제거라는 설계 목표에 따른 선택이다.

       콘텐츠 프래그먼트 vs 전체 문서: isFullDocument() 함수가 HTML이 <!doctype 또는 <html로 시작하는지 검사한다. 프래그먼트이면 frame-template.html로 감싸고(wrapInFrame), 전체 문서이면 그대로 서빙한다. helper.js는 어느 경우든 </body> 태그 앞에 주입된다.

       화면 전환: getNewestScreen()이 screen_dir 내 HTML 파일을 수정 시간(mtime) 기준으로 정렬해서 가장 최신 파일을 서빙한다. AI가 새 파일을 쓰면 자동으로 그것이 표시된다.

       이벤트 파일 초기화: 새 화면(새 파일)이 감지되면 .events 파일을 삭제한다. 이전 화면의 클릭 이벤트가 다음 화면으로 오염되는 것을 방지한다. 기존 파일 업데이트와 신규 파일을 knownFiles Set으로 구분한다.

       생명주기 관리:
       - 30분 비활동 시 자동 종료 (IDLE_TIMEOUT_MS)
       - 소유자 프로세스(AI 하네스) 종료 시 자동 종료 (ownerAlive() 검사, 60초 간격)
       - .server-info 파일로 서버 상태 공유, 종료 시 .server-stopped 파일 생성

       포트 할당: 임의의 high port (49152~65534)를 사용한다. 환경 변수 BRAINSTORM_PORT로 오버라이드 가능하다.

       5.3 helper.js (클라이언트 사이드)

       IIFE로 감싸진 클라이언트 코드로, 주요 기능:

       1. WebSocket 자동 연결/재연결: 연결이 끊기면 1초 후 자동 재시도. 오프라인 중 발생한 이벤트는 큐에 쌓았다가 연결 복원 시 전송한다.
       2. 선택 UI: data-choice 속성이 있는 요소 클릭을 캡처하여 서버로 전송한다. toggleSelect() 함수가 단일 선택과 다중 선택(data-multiselect)을 모두 처리한다.
       3. 인디케이터 바 업데이트: 선택 상태에 따라 하단 바의 텍스트를 동적으로 갱신한다 ("X selected -- return to terminal to continue").
       4. 공개 API: window.brainstorm.send()와 window.brainstorm.choice()로 커스텀 이벤트 전송 가능.

       5.4 frame-template.html

       OS의 다크/라이트 모드를 자동 감지하는 (prefers-color-scheme) CSS 변수 시스템을 갖춘 프레임 템플릿이다. 제공하는 CSS 컴포넌트:

       - .options / .option: A/B/C 텍스트 선택지 (세로 리스트)
       - .cards / .card: 비주얼 디자인 카드 (그리드 레이아웃)
       - .mockup: 와이어프레임 컨테이너
       - .split: 좌우 비교 뷰
       - .pros-cons: 장단점 표
       - .mock-nav, .mock-sidebar, .mock-button 등: 와이어프레임 빌딩 블록

       5.5 start-server.sh / stop-server.sh

       start-server.sh의 주요 설계 결정:

       - 환경 자동 감지: CODEX_CI 환경 변수가 있으면 자동으로 포그라운드 모드로 전환한다. Codex는 백그라운드 프로세스를 종료(reap)하기 때문이다.
       - 소유자 PID 추적: $PPID의 부모(즉, 스크립트의 조부모 프로세스)를 찾아서 실제 하네스 PID로 사용한다. 하네스가 스크립트를 실행하기 위해 생성한 일회성 셸($PPID)은 스크립트 종료 시 사라지므로, 그 부모를 추적해야 한다.
       - 서버 시작 확인: 로그 파일에서 server-started를 최대 5초간 폴링하며, 시작 후에도 프로세스가 살아있는지 2초간 추가 확인한다(프로세스 리퍼 대응).
       - 세션 격리: 각 세션이 고유한 디렉터리($$-$(date +%s))를 가져서 동시 실행 시 충돌하지 않는다.

       stop-server.sh는 /tmp 아래의 임시 세션만 삭제하고, .superpowers/ 아래의 영구 세션은 보존한다. 목업을 나중에 참조할 수 있도록 하기 위함이다.

       6. 도구(Tool)와 모드(Mode)의 구분

       Visual Companion에 대한 중요한 설계 철학: "도구이지 모드가 아니다." 사용자가 수락하더라도 모든 질문이 브라우저를 거치는 것이 아니다. 질문마다 "이것을 보여주는 것이 읽게 하는 것보다 나은가?"를 판단한다.

       구체적 기준:
       - 브라우저 사용: UI 목업, 아키텍처 다이어그램, 레이아웃 비교 등 시각적 콘텐츠
       - 터미널 사용: 요구사항 질문, 개념적 선택, 트레이드오프 리스트, 스코프 결정 등 텍스트 콘텐츠

       "UI 주제에 대한 질문이라고 자동으로 시각적 질문은 아니다"라는 예시가 이 구분을 명확히 한다.

       7. Superpowers 파이프라인에서의 위치

       Brainstorming 스킬은 Superpowers의 전체 개발 파이프라인에서 첫 번째 관문이다:

       brainstorming ──> writing-plans ──> executing-plans ──> verification-before-completion
         (설계)            (계획 수립)        (구현)              (검증)

       using-superpowers 스킬(메타 스킬)이 모든 대화의 시작점에서 "창작적 작업 전에 반드시 brainstorming을 사용하라"고 강제한다. SKILL.md의 description 필드 자체가 이를 명시한다:

       ▎ "You MUST use this before any creative work - creating features, building components, adding functionality, or modifying behavior."

       Brainstorming의 출력물은:
       8. docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md (설계 문서)
       9. 서브에이전트 리뷰를 통과한 검증된 스펙

       이 출력물이 writing-plans 스킬의 입력이 된다. writing-plans는 이 스펙을 받아 구현 계획을 작성한다.

       10. 주목할 만한 기술적 결정

       8.1 제로 의존성 서버

       server.js는 http, fs, path, crypto만 사용한다. WebSocket을 직접 구현한 이유는 npm install 없이 어떤 Node.js 환경에서든 바로 실행 가능하도록 하기 위함이다. AI 에이전트가 패키지 설치를 하는 것은 불필요한 부작용이므로, 이를 제거한 것이다.

       8.2 파일 시스템 기반 IPC

       AI 에이전트 <-> 서버 <-> 브라우저 간 통신에서, AI 에이전트와 서버 사이는 파일 시스템으로 소통한다. AI 에이전트는 대부분의 환경에서 파일 읽기/쓰기가 가능하지만 HTTP 클라이언트 호출은 제한될 수 있다. 파일 시스템이 가장 보편적인 공유 채널이라는 현실적 판단이다.

       8.3 디바운싱

       fs.watch 이벤트는 OS마다 다르게 동작한다. macOS에서는 파일 생성과 덮어쓰기 모두 rename 이벤트를 발생시킨다. 서버는 100ms 디바운스 타이머와 knownFiles Set을 조합하여 새 화면과 업데이트를 구분한다.

       8.4 크로스 플랫폼 서버 실행

       start-server.sh는 세 가지 주요 AI 하네스 환경에 대한 실행 전략을 갖추고 있다:
       - Claude Code: 기본 모드 (스크립트가 자체적으로 백그라운딩)
       - Codex: 환경 변수 자동 감지로 포그라운드 전환
       - Gemini CLI: --foreground 플래그 + 플랫폼의 백그라운드 실행 메커니즘 사용

       10. 결론

       Brainstorming 스킬은 단순한 "질문하고 답하기" 프로세스가 아니다. 이것은 AI 에이전트의 가장 흔한 실패 모드 -- 확인되지 않은 가정 위에 즉시 코드를 작성하는 것 -- 를 구조적으로 차단하는 시스템이다. Hard Gate, 단일 질문 원칙, 자동 스펙 리뷰, 파이프라인 단일 출구
       같은 제약들이 모두 같은 방향을 가리킨다: AI가 충분히 생각한 후에만 행동하도록 강제하기.

       Visual Companion은 이 철학의 연장선에서, 텍스트만으로 전달하기 어려운 시각적 의사결정을 지원하되, 제로 의존성과 파일 시스템 기반 통신이라는 현실적 제약 안에서 최대한의 유연성을 확보한 설계다.
