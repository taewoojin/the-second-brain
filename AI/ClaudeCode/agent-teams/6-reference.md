# 아키텍처와 제약

## 한눈에 보기

Agent Teams가 내부적으로 어떻게 동작하는지, 그리고 현재 버전의 한계가 무엇인지를 다룹니다. 깊이 있게 알고 싶을 때만 읽으세요.

---

## Agent Teams의 구조

### 주요 요소

```
팀 리더 (당신)
├─ 팀원 1 (독립된 Claude Code 인스턴스)
├─ 팀원 2 (독립된 Claude Code 인스턴스)
├─ 팀원 3 (독립된 Claude Code 인스턴스)
└─ 공유 자원
   ├─ 작업 목록
   ├─ 메시지함
   └─ 팀 설정 파일
```

**팀 리더**: 팀을 만들고 관리하는 당신. 팀원들에게 지시하고, 메시지를 받고, 최종 판단을 한다.

**팀원**: 각자 독립된 Claude Code 세션을 가지며, 자신의 컨텍스트 윈도우가 있다. 팀원들끼리 직접 메시지할 수 있다.

**공유 작업 목록**: 모든 팀원이 접근할 수 있고, 각 작업의 상태(대기→진행→완료)와 의존성을 관리한다.

**메시지함**: 팀원 간 메시지를 자동으로 전달한다.

## 파일 시스템 구조

Agent Teams를 만들면 두 개의 디렉토리가 생깁니다:

```
~/.claude/
├─ teams/
│  └─ {팀-이름}/
│     └─ config.json         (팀 설정, 팀원 정보)
└─ tasks/
   └─ {팀-이름}/
      ├─ task-1.json
      ├─ task-2.json
      └─ ...
```

### config.json 구조

```json
{
  "teamName": "my-team",
  "leadSessionId": "leader-123",
  "members": [
    {
      "name": "보안팀원",
      "agentId": "agent-456",
      "agentType": "security-reviewer"
    },
    {
      "name": "성능팀원",
      "agentId": "agent-789",
      "agentType": "performance-analyst"
    }
  ],
  "created": "2025-04-08T10:30:00Z"
}
```

config.json을 직접 수정하지 마세요. Claude Code가 자동으로 관리합니다.

## 팀원들의 컨텍스트

### 팀원이 받는 정보

1. **프로젝트 컨텍스트**
   - CLAUDE.md 파일 (프로젝트 규칙)
   - .claude/settings.json (설정)
   - MCP 서버 연결
   - 스킬

2. **팀 정보**
   - 팀 설정 읽기 권한 (다른 팀원이 누구인지 알기 위해)
   - 공유 작업 목록

3. **리더의 지시**
   - 팀 생성 시 받은 초기 프롬프트
   - 이후 수동 메시지

### 팀원이 받지 않는 정보

**리더의 과거 대화**: 팀원은 리더와의 과거 대화를 모르므로, 필요한 정보는 명시적으로 전달해야 합니다.

## 메시지 전달 방식

### 자동 메시지 전달

팀원이 메시지를 보내면:

```
팀원 → 메시지함 → 자동으로 리더에게 전달
```

특별히 할 일 없음. 자동입니다.

### 수동 메시지 (리더가 지시)

리더가 팀원에게 메시지를 보내면:

```
리더 → 메시지함 → 팀원에게 전달
```

### 팀원 간 메시지

팀원들끼리 필요하면 자동으로 메시지 교환:

```
팀원 A → 메시지함 → 팀원 B
```

리더는 알 수도 있고 모를 수도 있습니다. (투명성 수준에 따라)

## 작업 의존성 관리

### 의존성 설정

작업 A가 끝나야 작업 B를 시작할 수 있는 경우:

```
작업 B는 "작업 A에 의존함"으로 설정
```

### 자동 언블로킹

작업 A 완료 → 자동으로 작업 B가 "실행 가능"으로 변함

팀원들이 자동으로 감지하고 다음 작업을 선택합니다.

## 현재 버전의 제약

**세션 재개 미지원**: `/resume`로 팀 세션을 복구할 수 없으면 새 팀을 만들어야 한다.

**작업 상태 지연**: 팀원이 작업 완료를 표시하지 않으면 리더가 수동으로 표시해야 한다.

**종료 속도**: 팀원을 종료할 때 현재 작업이 끝날 때까지 기다린다.

**한 번에 한 팀**: 리더가 동시에 여러 팀을 관리할 수 없으므로 순차적으로 관리해야 한다.

**중첩 팀 불가**: 팀원은 하위 팀을 만들 수 없고, 평면 구조만 가능하다.

**리더 고정**: 팀을 만든 세션이 항상 리더이므로 리더를 바꿀 수 없다.

**권한은 생성 시점에 설정**: 팀원들은 생성 시점의 리더 권한을 상속받으므로, 이후 권한 변경은 기존 팀원에게 적용되지 않는다.

**Split pane 환경 제약**: VS Code, Windows Terminal, Ghostty에서는 split pane이 불가능하므로 in-process 모드를 사용해야 한다.

## 토큰 비용 계산

### 기본 계산

```
팀 비용 = 팀원 수 × 개별 Claude Code 비용
```

### 실제 예시

**3명 팀, 1시간 작업:**
- 팀원 1: 처리 30분 = 약 50,000 토큰
- 팀원 2: 처리 35분 = 약 58,000 토큰
- 팀원 3: 처리 25분 = 약 42,000 토큰
- 리더: 조정 및 종합 = 약 30,000 토큰

**총계**: 약 180,000 토큰 ≈ $2.70 (Claude 3.5 Sonnet 기준)

**혼자 할 경우:**
- 4시간 작업 = 약 200,000 토큰 ≈ $3.00

3명 팀이 4시간 일을 1시간에 끝내면, 비용은 10배 저렴하고 시간은 4배 단축된다.

## CLAUDE.md 파일과 팀

### 팀원이 CLAUDE.md를 읽나?

네, 각 팀원은 프로젝트의 CLAUDE.md를 자동으로 로드한다.

예를 들어, CLAUDE.md에 "모든 변경은 테스트가 있어야 한다"고 쓰면 팀원들이 이 규칙을 따른다.

### 팀 전용 지시사항

CLAUDE.md에 팀 전용 섹션을 만들 수 있습니다:

```markdown
## Agent Teams 가이드

팀원들에게 다음을 지켜달라고 말합니다:
- 각자 할당된 파일만 수정
- 변경 전에 물어보기
- 30분마다 진행 상황 보고
```

## Subagent 정의 활용

### Subagent를 팀원으로 쓰기

미리 정의한 subagent 타입을 팀원으로 사용할 수 있습니다:

```text
리더: "보안 리뷰 팀원을 만들어줄래"

→ ~/.claude/agents/security-reviewer.md의 정의를 따름
```

주의: Subagent의 `tools` 제한은 적용되지만, `skills`과 `mcpServers`는 적용되지 않는다. 기본 팀 조정 도구는 항상 사용 가능하다.

## 성능 최적화 팁

**팀원당 작업 수**: 5~6개가 적당하다. 너무 적으면 팀원이 기다리고, 너무 많으면 혼란스러워한다.

**첫 지시는 상세하게**: "리팩토링해줄래"보다는 "src/api/ 폴더의 중복된 유틸 함수를 정리해줄래. 특히 errorHandler-*.js 파일들이 비슷하니 통합할래?"라는 식으로 구체적으로.

**진행 확인**: 매 30분마다 진행 상황을 물어보자.

**방향 수정**: 팀원이 잘못된 방향으로 가면 빨리 피드백을 준다.
