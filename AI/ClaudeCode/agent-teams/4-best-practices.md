# 실무 팁과 사례

## 한눈에 보기

팀으로 일할 때 알아두면 좋은 팁들입니다. 이는 규칙이 아니라 경험으로부터 나온 효율적인 방법들입니다.

---

## 팀 크기 정하기

### 최적의 팀 크기

**3~5명** 정도가 가장 잘 작동합니다. 1~2명이면 팀의 이점이 없고, 6명 이상이면 의사소통 오버헤드와 토큰 비용이 늘어나며 조정이 복잡해집니다.

### 팀원당 작업 수

각 팀원이 5~6개의 독립적인 작업을 하면 효율적입니다. 15개 작업이면 3명 팀, 20개면 4명, 30개면 5명 정도가 맞습니다.

---

## 작업 분배 전략

### 전략 1: 영역으로 나누기

각 팀원이 다른 파일이나 폴더를 담당합니다:

```
팀원 1: 프론트엔드 컴포넌트
팀원 2: 백엔드 API
팀원 3: 테스트 코드
```

**장점**: 팀원들이 같은 파일을 건드리지 않음  
**단점**: 파일 영역이 명확해야 한다

### 전략 2: 관점으로 나누기

같은 코드를 다른 관점에서 검토합니다:

```
팀원 1: PR 보안 검토
팀원 2: PR 성능 검토
팀원 3: PR 테스트 커버리지 검토
```

**장점**: 깊이 있는 검토 가능  
**단점**: 각 팀원이 전체를 이해해야 함

### 전략 3: 가설로 나누기

버그 원인을 여러 팀원이 동시에 조사합니다:

```
팀원 1: "데이터 로딩 문제일 수도 있어"
팀원 2: "메모리 누수일 수도 있어"
팀원 3: "타이밍 문제일 수도 있어"
```

**장점**: 가설을 병렬로 검증할 수 있다  
**단점**: 한 팀원이 답을 찾으면 다른 팀원들의 노력이 중복될 수 있다

## 자주 하는 실수와 해결법

### 실수 1: 팀원이 준비 안 된 채 시작

**문제**: "코드를 리팩토링해줄래"라고만 말하고 자세한 설명 없음  
**결과**: 팀원이 뭘 기준으로 할지 몰라 시간이 낭비된다

**해결법**:
```text
(좋은 예)
API 응답 객체를 정리해줄래? 
지금 src/types/api.ts에 타입 200개가 너무 많아.
중복되는 게 있을 거 같으니 정리해서 50개 정도로 줄여줄 수 있을까?
```

### 실수 2: 팀원끼리 같은 파일 수정

**문제**: A 팀원과 B 팀원이 같은 파일 수정  
**결과**: 한쪽 변경이 덮어씌워짐

**해결법**: 처음부터 다른 파일로 분리하거나, 한 팀원만 수정하도록 할당한다

### 실수 3: 팀 리더가 먼저 작업 시작

**문제**: 리더가 "내가 할게"라고 팀원들 무시하고 시작  
**결과**: 팀원들이 기다리거나 충돌이 발생한다

**해결법**:
```text
"팀원들이 그 부분 끝날 때까지 기다려줄래"
```

### 실수 4: 팀원 대기 시간 너무 김

**문제**: 팀원이 오래 기다리게 됨  
**결과**: 시간이 오래 걸리거나 팀원이 방향 잃음

**해결법**: 자주 체크인하고, 팀원이 다른 일을 할 수 있게 하며, 필요하면 방향을 수정한다

## Agent Teams가 빛나는 사례

### 사례 1: PR 병렬 리뷰

```
상황: 중요한 PR 검토 필요, 빨리 병합해야 함

팀 구성:
- 리더: 최종 판단
- 보안팀원: 보안 취약점 찾기
- 성능팀원: 성능 영향 분석
- 테스트팀원: 테스트 충분도 확인

결과: 1시간 vs 4시간 (혼자)
```

### 사례 2: 버그 원인 찾기

```
상황: 앱이 간헐적으로 종료됨 (원인 불명확)

팀 구성:
- 리더: 증거 정리
- 팀원 1: "메모리 누수일 거"라고 가정하고 조사
- 팀원 2: "경합 조건 문제일 거"라고 가정하고 조사
- 팀원 3: "외부 API 응답 오류일 거"라고 가정하고 조사

결과: 팀원 3이 원인 발견! (실제 버그는 API 타임아웃)
```

### 사례 3: 문서 동시 작성

```
상황: 50페이지 API 문서 작성 (한 명이면 1주)

팀 구성:
- 팀원 1: 인증 섹션
- 팀원 2: 사용자 API 섹션
- 팀원 3: 결제 API 섹션

결과: 2~3일이면 완성, 리더가 최종 정리
```

## 효율성 체크리스트

팀을 만들기 전에 확인하세요:

- ✓ 여러 팀원이 동시에 일할 수 있는가?
- ✓ 각 팀원이 다른 파일/영역을 담당하는가?
- ✓ 팀원들의 작업이 서로 기다리지 않아도 되는가?
- ✓ 충분한 배경 정보를 제공할 수 있는가?

하나라도 해당하지 않으면 팀을 쓰기보다 다른 방법이 나을 수 있습니다.
