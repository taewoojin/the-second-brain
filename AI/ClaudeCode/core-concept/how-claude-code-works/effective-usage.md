# 효과적인 사용법

> **한눈에 보기**
> Claude Code는 대화형으로 작동하므로 완벽한 프롬프트가 필요 없다. 구체적으로 요청할수록 수정 횟수가 줄고, 검증 기준을 함께 주면 결과가 좋아진다. 복잡한 문제는 탐색과 구현을 분리하고, 세부 사항은 Claude에게 위임한다.

## Claude Code에게 도움 요청하기

Claude Code 자체 사용법을 Claude Code에게 물어볼 수 있다. "훅은 어떻게 설정해?", "CLAUDE.md 구조를 어떻게 짜는 게 좋아?" 같은 질문에 Claude가 설명해준다.

내장 명령어로도 설정을 안내받을 수 있다.

| 명령어 | 역할 |
|--------|------|
| `/init` | 프로젝트용 CLAUDE.md 생성을 안내 |
| `/agents` | 커스텀 서브에이전트 설정을 안내 |
| `/doctor` | 설치 관련 일반적인 문제를 진단 |

## 대화하듯 사용한다

완벽한 프롬프트를 작성할 필요가 없다. 원하는 것을 말하고, 결과를 보고, 다시 조정하면 된다.

```
로그인 버그 고쳐줘
```

[Claude가 조사하고 시도한다]

```
그건 아닌데. 세션 핸들링 쪽이 문제야.
```

[Claude가 접근 방식을 조정한다]

첫 시도가 틀려도 처음부터 다시 시작할 필요가 없다. 대화를 이어가면 된다.

### 중간에 끼어들기

Claude가 잘못된 방향으로 가고 있으면, 수정 내용을 입력하고 Enter를 누르면 된다. Claude는 하던 것을 멈추고 입력에 맞게 접근 방식을 조정한다. 끝날 때까지 기다리거나 처음부터 다시 시작할 필요가 없다.

## 처음부터 구체적으로 요청한다

초기 프롬프트가 정확할수록 수정 횟수가 줄어든다. 특정 파일을 언급하고, 제약 조건을 명시하고, 예시 패턴을 가리킨다.

```
만료된 카드를 가진 사용자의 결제 플로우가 깨져 있어.
src/payments/를 확인해봐, 특히 토큰 갱신 쪽.
실패하는 테스트를 먼저 작성하고, 그 다음에 수정해.
```

모호한 프롬프트도 작동하지만 방향 수정에 더 많은 시간을 쓰게 된다. 위처럼 구체적인 프롬프트는 첫 시도에 성공하는 경우가 많다.

## 검증 기준을 함께 준다

Claude는 자기 작업을 스스로 확인할 수 있을 때 더 좋은 결과를 낸다. 테스트 케이스를 포함하거나, 기대하는 UI의 스크린샷을 붙여넣거나, 원하는 출력을 정의한다.

```
validateEmail을 구현해. 테스트 케이스:
'user@example.com' → true, 'invalid' → false, 'user@.com' → false.
끝나면 테스트 실행해.
```

시각적 작업에서는 디자인 스크린샷을 붙여넣고 구현 결과와 비교하도록 요청할 수 있다.

## 탐색과 구현을 분리한다

복잡한 문제에서는 리서치와 코딩을 나눈다. 플랜 모드(`Shift+Tab` 두 번)로 먼저 코드베이스를 분석한다.

```
src/auth/를 읽고 세션 처리 방식을 파악해.
그 다음에 OAuth 지원 추가 계획을 세워.
```

계획을 검토하고, 대화로 다듬은 뒤, Claude에게 구현하게 한다. 이 두 단계 접근이 바로 코드로 뛰어드는 것보다 더 나은 결과를 만든다.

## 위임하되, 지시하지 않는다

유능한 동료에게 일을 맡기듯이 한다. 맥락과 방향을 주고, 세부 사항은 Claude가 알아서 하도록 맡긴다.

```
만료된 카드를 가진 사용자의 결제 플로우가 깨져 있어.
관련 코드는 src/payments/에 있어. 조사하고 고쳐줄 수 있어?
```

어떤 파일을 읽을지, 어떤 명령을 실행할지 일일이 지정할 필요가 없다. Claude가 스스로 파악한다.
