# 어떤 기능을 쓸까?

> **한눈에 보기**
> "항상 알아야 하는 것"은 CLAUDE.md, "가끔 필요한 지식이나 워크플로"는 Skills, "외부 연결"은 MCP, "컨텍스트 격리"는 Subagents, "팀 협업"은 Agent Teams, "자동 실행"은 Hooks. 패키징은 Plugins.

---

## Skill vs Subagent

둘 다 "본체와 분리된 무언가"처럼 보이지만, 해결하는 문제가 다르다.

- **Skill**은 재사용 가능한 콘텐츠(지식 또는 워크플로)다.
- **Subagent**는 격리된 실행 환경이다.

| 관점 | Skill | Subagent |
|------|-------|----------|
| **본질** | 마크다운 문서 (지식 + 절차) | 독립 워커 (자체 컨텍스트) |
| **핵심 이점** | 여러 맥락에서 동일 콘텐츠 재사용 | 메인 대화를 오염시키지 않는 격리 |
| **적합한 상황** | API 가이드, 배포 체크리스트 | 대량 파일 분석, 병렬 작업 |

**조합도 가능하다:** Subagent 생성 시 특정 Skills를 미리 로드할 수 있고, Skill의 `context: fork` 옵션으로 격리된 컨텍스트에서 실행할 수도 있다.

**선택 기준:**
- "이 정보를 Claude가 알아야 한다" → Skill
- "이 작업을 메인 대화 밖에서 처리하고 결과만 받고 싶다" → Subagent

---

## CLAUDE.md vs Skill

둘 다 "Claude에게 지시를 전달하는 마크다운"이지만, 로딩 방식과 용도가 다르다.

| 관점 | CLAUDE.md | Skill |
|------|-----------|-------|
| **로딩 시점** | 매 세션 시작 시 자동 | 사용할 때만 (이름·설명은 항상 로드) |
| **워크플로 실행** | 불가 | `/이름`으로 호출 가능 |
| **적합한 내용** | "항상 이렇게 해라" 규칙 | 참고 자료, 반복 워크플로 |

**선택 기준:**
- "Claude가 항상 알아야 한다" → CLAUDE.md
- "가끔 참고하거나, 특정 명령으로 실행한다" → Skill

CLAUDE.md가 200줄을 넘으면 참고 자료는 Skills로, 경로별 규칙은 `.claude/rules/`로 분리한다.

---

## CLAUDE.md vs Rules vs Skill

세 가지 모두 지시를 저장하지만, 로딩 범위가 다르다.

| 관점 | CLAUDE.md | `.claude/rules/` | Skill |
|------|-----------|-------------------|-------|
| **로딩** | 매 세션 | 매 세션 또는 해당 파일 작업 시 | 사용 시 |
| **범위** | 프로젝트 전체 | 파일 경로로 한정 가능 | 작업별 |
| **적합한 내용** | 핵심 컨벤션, 빌드 명령어 | 언어별·디렉토리별 가이드라인 | 참고 자료, 반복 워크플로 |

**선택 기준:**
- 모든 세션에 필요한 핵심 규칙 → CLAUDE.md
- 특정 파일/디렉토리 작업 시에만 필요한 규칙 → Rules
- 특정 작업에만 필요한 지식이나 절차 → Skill

## Subagent vs Agent Teams

둘 다 작업을 병렬화하지만, 구조적으로 다르다.

| 관점 | Subagent | Agent Teams |
|------|----------|-------------|
| **비유** | 상사에게 보고하는 워커 | 동료끼리 논의하는 팀 |
| **소통 방식** | 결과를 메인 에이전트에게만 반환 | 팀원끼리 직접 메시지 교환 |
| **조율 방식** | 메인 에이전트가 모든 작업 관리 | 공유 태스크 리스트로 자율 조율 |
| **토큰 비용** | 낮음 (결과 요약만 메인에 전달) | 높음 (각 팀원이 독립 세션) |

**선택 기준:**
- "작업을 시키고 결과만 받으면 된다" → Subagent
- "팀원들이 서로 발견한 것을 공유하고, 판단에 이의를 제기해야 한다" → Agent Teams

**전환 시점:** 병렬 Subagent를 사용하다 컨텍스트 한계에 부딪히거나, Subagent끼리 소통이 필요해지면 Agent Teams로 전환을 고려한다.

## MCP vs Skill

이 둘은 사실 비교 대상이 아니라 보완 관계다.

| 관점 | MCP | Skill |
|------|-----|-------|
| **본질** | 외부 서비스 연결 프로토콜 | 지식과 워크플로 |
| **제공하는 것** | 도구와 데이터 접근 | 지식, 절차, 참고 자료 |
| **예시** | DB 연결, Slack 전송, 브라우저 제어 | 코드 리뷰 체크리스트, 배포 절차, API 가이드 |

**MCP**는 Claude에게 외부 시스템과 상호작용하는 **능력**을 준다.
**Skill**은 그 능력을 효과적으로 사용하는 **방법**을 알려준다.

**조합 예시:** MCP 서버로 데이터베이스에 연결하고, Skill에 데이터 모델·주요 쿼리 패턴·테이블별 용도를 문서화한다. Claude는 MCP로 쿼리를 실행하되, Skill의 지식을 바탕으로 올바른 테이블과 패턴을 선택한다.

## 상황별 빠른 선택

| 하고 싶은 것 | 사용할 기능 |
|-------------|------------|
| 코딩 컨벤션을 매번 알려주기 귀찮다 | CLAUDE.md |
| Python 파일 작업 시에만 적용할 규칙이 있다 | `.claude/rules/` (paths 지정) |
| 배포 절차를 명령 한 번으로 실행하고 싶다 | Skill (`/deploy`) |
| API 스타일 가이드를 Claude가 참고하게 하고 싶다 | Skill (참고 자료형) |
| 많은 파일을 분석하되 메인 대화를 깨끗하게 유지하고 싶다 | Subagent |
| 보안·성능·스타일 리뷰를 동시에 돌리고 싶다 | Agent Teams |
| 데이터베이스를 조회하고 싶다 | MCP |
| 파일 저장 후 자동으로 린트를 돌리고 싶다 | Hooks |
| 위 기능들을 묶어서 다른 프로젝트에도 쓰고 싶다 | Plugins |
